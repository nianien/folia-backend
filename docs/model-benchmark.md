# 本地模型横向测评报告

对 `qwen3.6:35b-a3b`、`qwen3.5:9b`、`qwen3:14b`、`gemma4:12b`、`gemma3:4b` 五个本地
Ollama 模型，在 folia 流水线的三个真实 LLM 任务上做的横向对比。

> `qwen3.6:35b-a3b` 是后补测的；`a3b` = MoE 结构，35B 总参数但每 token 只激活 ~3B，
> 因此「质量按 35B、算力按 3B」，用完全相同的输入补跑，结果并入下表。

## 测试方法

- **任务与 prompt**：直接复用 `src/folia/pipeline/prompts.py` 的生产 prompt——
  `categorize`（新闻分类）、`facts`（结构化事实抽取，严格 JSON）、
  `synthesis`（多源成稿，带来源引用，中/英各一篇）。
- **请求参数**：与 `model_client._ollama` 一致——`think:False`、`temperature=0.2`、
  `num_predict=3000`；**`num_ctx=4096`**（见下方「重要环境结论」）。
- **输入**：数据库里 3 篇真实文章做分类、2 篇做事实抽取；成稿用同一份
  「加拿大选定潜艇供应商」双来源事实包（source 1 = 决策报道，source 2 = 结果报道），
  四个模型吃完全相同的输入，保证可比。
- **计时**：取 Ollama 返回的 `eval_count / eval_duration` 算 tokens/s；全程 `caffeinate -i`
  防止机器休眠污染墙上时间。
- **机器**：32GB 内存的 Mac，全部模型 100% 跑在 GPU。

## 性能（速度）

| 模型 | 生成速度 (tok/s) | 事实抽取墙上时间 | 成稿墙上时间 | 分类响应 | 常驻内存 |
|---|---|---|---|---|---|
| **qwen3.6:35b-a3b** | **42.6** | 13 / 14 s | 14–17 s | 0.7 s | 23 GB |
| **gemma3:4b** | 28.6 | 25 / 30 s | 15–16 s | 0.8 s | 3.3 GB |
| **qwen3.5:9b** | 16.7 | 42 / 50 s | 27–30 s | 1.1 s | ~6.6 GB |
| **gemma4:12b** | 10.3 | 70 / 88 s | 46–47 s | 1.8 s | 8 GB |
| **qwen3:14b** | 9.7 | 47 / 61 s | 54–57 s | 1.1 s | 10 GB |

速度排序：**qwen3.6:35b-a3b > gemma3:4b ≫ qwen3.5:9b > gemma4:12b ≈ qwen3:14b**。
`qwen3.6:35b-a3b` 靠 MoE（每 token 仅激活 ~3B）拿下最快生成速度，比稠密的 9B/14B 还快，
但**代价是 23GB 常驻内存**（详见下方环境结论）。gemma3:4b 次之，约为两个 12B/14B 的
**3 倍**快。分类任务只输出几个 token，五个模型都在 1~2 秒内完成，差异可忽略。

## 效果（质量）

四个模型在三类任务上的输出都能被流水线正确解析、无致命格式错误。细节差异：

**categorize（分类）**
- 三篇测试文章（卫星救援 / 加拿大潜艇 / 乌克兰克里米亚）五个模型都分类正确
  （`科技` / `国际` / `国际`）。
- 除 gemma3:4b 一次输出了 `科技/国际`——目录里 `国际` 并非 `科技` 的二级分类，
  属于「自造二级分类」，违反了 prompt 规则。其余模型（含 qwen3.6:35b-a3b）都严格单行、
  只用目录内名称。

**facts（事实抽取）**——全部输出合法 JSON、字段齐全、元数据原样返回：

| 模型 | 抽取事实条数 | type 枚举合规 | JSON 包裹 |
|---|---|---|---|
| gemma4:12b | 14–15（最丰富） | 偶发 `background`（越界） | ` ```json ` 代码块 |
| qwen3.5:9b | 12–15 | 偶发 `background`（越界） | 裸 JSON |
| gemma3:4b | 9（偏少） | 合规 | 裸 JSON |
| qwen3:14b | 7–8（最保守） | 合规 | 裸 JSON |
| qwen3.6:35b-a3b | 6–7（高度合并） | 合规 | 裸 JSON |

- **信息量**：gemma4:12b 与 qwen3.5:9b 抽得最全；qwen3:14b 最保守/精简，容易漏细节。
  qwen3.6:35b-a3b 条数看似最少，但属**高质量合并**——把「今年达到 2% → 2035 前提到 5%」
  这类关联事实压成一条，关键数字（12 艘、5%、2035、2%、四艘维多利亚级、90 年代末）
  与直接引语一个不漏，是「精炼」而非「遗漏」。
- **schema 遵守**：qwen3.5 与 gemma4 偶尔把 `background` 当成 `type` 值（允许集合是
  event/decision/statement/cause/impact/timeline），属轻微 schema drift；
  qwen3:14b、gemma3:4b、qwen3.6:35b-a3b 全程合规。
- gemma4 的 ` ```json ` 代码块**已被 `facts.py:parse_json_object` 兼容剥离**，不影响入库。

**synthesis（成稿）**——四个模型都产出合法 Markdown、正确用 `[1][2]` 引用、
多源事实做了去重合并、未编造超出 {1,2} 的来源、未擅自输出 Sources 段：

- **qwen3.6:35b-a3b 中文稿质量最佳**：945 字、24 处引用，结构最清晰（关键细节下分
  「合同/竞标、舰队更新、国防支出、时间线」小节），忠实、去重、无杜撰，还准确还原了
  「原计划五月公布、实际提前到周一」这类时间线细节。
- qwen3.5:9b / qwen3:14b / gemma4:12b 引用密度高（14–18 处）、结构完整
  （标题 + 关键细节 + 背景），忠实无杜撰。qwen3.5:9b 的中文稿在中型模型里最好。
- gemma3:4b 中文稿偏薄（547 字、仅 5 处引用，引用密度明显不足），英文稿正常
  （17 处引用）——小模型在中文长文成稿上的稳定性最弱。

## 重要环境结论（生产隐患）

测评过程中发现一个**与模型无关、但影响生产**的问题：

- `model_client._ollama` 只设了 `temperature`/`num_predict`，**没设 `num_ctx`**，
  Ollama 于是对这些模型默认开 **32K 上下文**。在 32GB 内存的机器上，12B+ 权重
  + 32K KV cache 会超出可用统一内存，generate 阶段溢出到 CPU/swap，
  实测掉到 **~0.4 tok/s**（一条事实抽取要十几分钟到一个多小时），实际不可用。
- 把 `num_ctx` 压到 **8192**（足够覆盖本项目最坏 ~2K 输入 + 3K 输出）后，
  五个模型全部稳定留在 GPU，上表数据即为该配置下结果。
- KV cache 随 `num_ctx` 线性增长。实测 qwen3:14b：`num_ctx=4096` 常驻 10GB，
  `num_ctx=32768` 常驻 14GB——仅上下文一项就多吃 ~4GB。
- **`qwen3.6:35b-a3b` 是这条修复的最大受益者**：它权重就占 23GB，默认 32K 上下文的
  KV cache 会直接把 32GB 内存顶爆、退化到 swap；限到 8192 后总常驻 23GB、100% GPU，
  反而是全场最快。
- **已落地**：`settings.model` 加了 `num_ctx`（默认 8192），`model_client._ollama` 在
  `options` 里下发（仅 Ollama，远程 provider 不受影响）；可经 settings 表覆盖。

（另注：后台长任务按真实墙上时间计时，机器休眠期间 GPU 推理被挂起、
Ollama 把冻结时间计入 `eval_duration`，会造成「假性极慢」；最终数据均在
`caffeinate` 防休眠下测得。）

## 结论与选型建议

补测后格局变了——**`qwen3.6:35b-a3b` 是「质量最高 + 速度最快」的双料第一**：
分类全对、事实抽取合并精炼且 type 全合规、中文成稿质量与引用密度全场最好，
生成 42.6 tok/s 也是最快。唯一代价是 **23GB 常驻内存**，在 32GB 机器上余量薄、
且不能与其他大模型同时留驻。

按能否接受这 23GB 占用，给两套推荐：

**A. 内存够、追求最佳效果（推荐）：三个功能都用 `qwen3.6:35b-a3b`。**
质量与速度都最优，且三功能共用一份常驻，不会重复占内存。前提是 `num_ctx` 已限到 8192
（已落地）、且别再给别的功能配其他大模型。

**B. 想留内存 / 更保守：以 `qwen3.5:9b` 为默认，按功能细分：**

| 功能 | 首选 | 理由 |
|---|---|---|
| categorize | gemma3:4b | 高频、只输出一行，最快最省；注意个别自造二级分类 |
| facts | qwen3.5:9b | 抽取全、速度可用、裸 JSON；gemma4 更全但慢近一倍 |
| synthesis | qwen3.5:9b | 忠实、引用规范、中文稿最好；追求更长英文稿可选 qwen3:14b |

各模型定位：
- **qwen3.6:35b-a3b**：MoE 带来「35B 质量、3B 速度」，全场最佳；只受内存约束。
- **qwen3.5:9b**：中型模型里质量/速度平衡最好，6.6GB 占用友好，稳妥的默认。
- **gemma3:4b**：速度快、体积仅 3.3GB，适合高频/低延迟；代价是抽取偏少、中文长稿偏薄、
  需盯紧分类越界。
- **gemma4:12b**：事实抽取最丰富，但慢、输出带代码块；性价比不突出。
- **qwen3:14b**：成稿文笔不错，但抽取最保守、速度最慢；本机性价比最低。

一句话：**内存够就三功能全上 `qwen3.6:35b-a3b`（最快又最好，靠 MoE 和已落地的
num_ctx=8192）；想省内存就默认 `qwen3.5:9b`、分类换 `gemma3:4b`。两个 12B/14B
在这台 32GB 机器上都不值得——更慢，质量没成比例领先。**
